\section{Revue de littérature}
\label{sec:litterature}

\subsection{Contexte et problématique}

La cartographie des cultures agricoles à grande échelle est un enjeu majeur
pour la gestion des ressources alimentaires et le suivi environnemental.
Les images satellitaires multi-temporelles offrent une couverture spatiale
importante, mais leur exploitation nécessite des modèles capables de traiter
des séries temporelles bruyantes (nuages, données manquantes) sur de grandes
surfaces.

\subsection{Analyse du papier MCTNet}

MCTNet (\textit{Multi-stage CNN-Transformer Network}) \cite{wang2024} propose
une architecture légère combinant CNN et Transformer pour la classification
pixel par pixel de cultures agricoles à partir de séries temporelles
Sentinel-2. Les contributions principales du papier sont :

\begin{itemize}
  \item \textbf{Architecture hybride CNN-Transformer} : un bloc CTFusion traite
        la série temporelle en deux branches parallèles — une branche CNN pour
        les motifs locaux, une branche Transformer pour les dépendances
        long-terme — et fusionne leurs sorties par concaténation.
  \item \textbf{Légèreté} : moins de 57 000 paramètres, contre plusieurs
        millions pour les modèles concurrents (TransUNet, Swin Transformer),
        ce qui rend MCTNet déployable sans GPU haut de gamme.
  \item \textbf{Gestion des données manquantes} : le module ALPE
        (\textit{Adaptive Learnable Positional Encoding}) pondère l'encodage
        positionnel en fonction d'un masque de disponibilité, atténuant
        l'impact des observations occultées par des nuages.
  \item \textbf{Performances} : OA de 0.9598 sur Arkansas et 0.9502 sur
        California, comparables aux modèles nettement plus lourds.
\end{itemize}

\subsection{Positionnement}

Par rapport aux approches existantes, MCTNet se distingue par son ratio
performance/complexité. Les modèles purement CNN (TempCNN \cite{pelletier2019})
capturent bien les motifs locaux mais peinent sur les dépendances long-terme.
Les modèles purement Transformer sont puissants mais coûteux. MCTNet propose
un compromis efficace en combinant les deux à chaque stage.

\section{Acquisition des données}
\label{sec:acquisition}

\subsection{Source et régions d'étude}

Les données Sentinel-2 Level-2A ont été acquises via \textbf{Google Earth
Engine} pour l'année 2021 sur deux régions agricoles américaines aux
profils très différents :

\begin{itemize}
  \item \textbf{Arkansas} — état du sud, dominé par des cultures annuelles
        en rotation (Corn, Cotton, Rice, Soybeans). Climat humide subtropical
        avec une forte couverture nuageuse en hiver (taux de données
        manquantes : 23.9\,\%, dont 3 pas de temps entièrement absents).
  \item \textbf{California} — état de l'ouest, caractérisé par des cultures
        pérennes (Grapes, Almonds, Pistachios) et irriguées (Rice, Alfalfa).
        Climat méditerranéen plus favorable à l'observation satellite (taux
        de données manquantes : 16.1\,\%).
\end{itemize}

Les labels de référence proviennent du \textbf{Cropland Data Layer (CDL)}
de l'USDA (résolution 30\,m, confiance $\geq 95\,\%$), validés par le masque
\textbf{ESA WorldCover 2021} (classe Cropland).

\subsection{Stratégie de zonage}

Engine. La solution adoptée consiste à découper chaque état en petites
bounding boxes ($\approx 0.5° \times 0.4°$) ciblant les zones agricoles
connues, exporter chaque zone séparément, puis fusionner en Python.

\subsection{Format des données}

Chaque pixel est décrit par trois tenseurs :

\begin{itemize}
  \item \texttt{input1} : $(N,\,10,\,36)$ — réflectances normalisées
        ($\div\,10\,000$, puis les valeurs 0 indiquent un pas de temps manquant).
  \item \texttt{input2} : $(N,\,36)$ — masque binaire construit avant
        normalisation : 1 si au moins une bande est non nulle, 0 sinon.
  \item \texttt{labels} : $(N,)$ — indice de classe entier.
\end{itemize}

\subsection{Dimension temporelle}

L'année 2021 est découpée en \textbf{36 fenêtres de 10 jours}. Pour
chaque fenêtre, la \textbf{médiane} des observations valides (sans nuages)
est calculée pixel par pixel. La médiane est préférée à la moyenne pour
sa robustesse aux nuages non détectés (valeurs aberrantes) :

\begin{equation}
  \text{composite}_t = \text{median}\!\big\{x_i \;\mid\; x_i \text{ valide dans } [t, t+10\text{j}]\big\}
\end{equation}

\section{Prétraitement}
\label{sec:preprocessing}

\subsection{Construction du masque (Input 2)}

Le masque est construit \textbf{avant} toute normalisation. Un pas de
temps est considéré manquant si toutes ses bandes valent simultanément 0
(convention GEE pour les fenêtres sans observation valide) :

\begin{equation}
  \text{mask}_{n,t} =
  \begin{cases}
    0 & \text{si } \sum_{b=1}^{10} x_{n,b,t} = 0 \\
    1 & \text{sinon}
  \end{cases}
\end{equation}

Aucune interpolation n'est appliquée sur les valeurs manquantes : les 0
sont conservés dans \texttt{input1} et c'est le module ALPE qui se charge
de les atténuer lors du traitement.

\subsection{Normalisation}

Sentinel-2 stocke les réflectances sous forme d'entiers multipliés par
10\,000. La normalisation divise par cette constante pour ramener les
valeurs dans $[0, 1]$ :

\begin{equation}
  \hat{x}_{n,b,t} = \frac{x_{n,b,t}}{10\,000}
\end{equation}

Les valeurs 0 (données manquantes) restent à 0 après normalisation, ce
qui est voulu. Cette opération est appliquée identiquement sur train, val
et test, sans fuite de données.

\subsection{Restructuration en tenseurs}

Les CSV exportés depuis GEE sont en format plat (360 colonnes par ligne,
nommées selon la convention \texttt{B2, B2\_1, ..., B12\_35}). La
restructuration en tenseurs \textit{channels-first} suit :

\begin{equation}
  (N, 360) \;\xrightarrow{\text{reshape}}\; (N, 36, 10)
            \;\xrightarrow{\text{transpose}}\; (N, 10, 36)
\end{equation}

Le format $(N, 10, 36)$ est le format \textit{channels-first} attendu par
les convolutions 1D de PyTorch.

\section{Architecture de MCTNet}
\label{sec:architecture}

\subsection{Vue d'ensemble}

MCTNet enchaîne trois blocs CTFusion suivis d'un Global Max Pooling et d'un
classificateur linéaire :

\begin{equation}
  (B,\,10,\,36)
  \xrightarrow{\text{Stage 1}} (B,\,20,\,18)
  \xrightarrow{\text{Stage 2}} (B,\,40,\,9)
  \xrightarrow{\text{Stage 3}} (B,\,80,\,4)
  \xrightarrow{\text{GMP}} (B,\,80)
  \xrightarrow{\text{Linear}} (B,\,N_c)
\end{equation}

$N_c = 5$ pour l'Arkansas, $N_c = 6$ pour la Californie.

\subsection{Bloc CTFusion}

Le bloc CTFusion est l'unité de base répétée trois fois. Il traite la même
entrée en deux branches parallèles puis fusionne leurs sorties :

\begin{itemize}
  \item \textbf{Branche CNN} (ResBlock1D) — deux convolutions 1D de kernel 3
        avec connexion résiduelle, capte les motifs locaux de la série
        temporelle (pics phénologiques, transitions rapides).
  \item \textbf{Branche Transformer} (MHSA, 5 têtes) — modélise les
        dépendances globales entre pas de temps distants.
\end{itemize}

Les sorties sont concaténées sur la dimension canaux, puis réduites
temporellement par un MaxPool1D (stride 2) :

\begin{equation}
  \text{CTFusion}(x) =
  \text{MaxPool1D}\!\Big(\big[\,\text{CNN}(x),\;\text{Transformer}(x)\,\big]\Big)
  \;:\; (B,C,T) \to (B,\,2C,\,T/2)
\end{equation}

\subsection{Module ALPE}

Au Stage 1 uniquement, un module ALPE (\textit{Adaptive Learnable Positional
Encoding}) pondère l'encodage positionnel du Transformer en fonction du masque
\texttt{input2} :

\begin{equation}
  \text{PE\_adapté}_t = \text{PE}_t \cdot w_t, \quad
  w_t = f\!\big(\text{input2}_t,\; \theta_{\text{ALPE}}\big)
\end{equation}

Les pas de temps couverts par des nuages ($\text{input2}_t = 0$) reçoivent
un encodage réduit, limitant leur influence sur les scores d'attention. Cette
approche est plus robuste que l'interpolation car elle n'introduit pas de
valeurs synthétiques dans les données.
